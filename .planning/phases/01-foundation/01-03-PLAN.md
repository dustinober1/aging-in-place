<objective>
Build per-category permission management with key rotation — enabling a senior to grant and revoke access to specific data categories for each care team member.

Purpose: Permission revocation with key rotation is the core privacy mechanism. New records in a revoked category become unreadable to the revoked member because the encryption key changes.
Output: Member detail view with permission toggles, undo toast on revocation, and background key rotation.
</objective>

  <behavior>
    PermissionTests (using in-memory ModelContainer):
    - Granting a category adds it to member.grantedCategories (TEAM-05)
    - Revoking a category removes it from member.grantedCategories (TEAM-06)
    - After revoking and rotating key: seal() with new key produces ciphertext that cannot be opened with key snapshot taken before rotation (TEAM-07)
    - Default new member has all 4 categories granted
    - Revoking last category leaves empty grantedCategories array
    - Granting an already-granted category is idempotent
  </behavior>
  <action>
    Write PermissionTests using XCTest with in-memory ModelContainer. Tests should:

    1. Create a CareCircle and CareTeamMember with default permissions
    2. Verify all 4 categories are granted by default
    3. Revoke a category → verify it's removed from grantedCategories
    4. Grant it back → verify it's added
    5. Test key rotation integration: snapshot key before revocation (seal some test data), revoke + rotate, try to open new sealed data with old key snapshot → should fail

    For the key rotation integration test, since we can't easily "snapshot" a Keychain key, test the EncryptionService behavior:
    - Seal data with current key for `.medications`
    - Call `EncryptionService.rotateKey(for: .medications)`
    - Seal new data — the new ciphertext should NOT be decryptable by opening with the old sealed data's pattern (this proves the key changed)
    - The old sealed data should also fail to open (old key was overwritten)

    Clean up Keychain in tearDown.
  </action>
